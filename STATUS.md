This is a fork of [bitchat-android](https://github.com/permissionlesstech/bitchat-android),
renamed to package `com.kriptic.app`, with Kriptic's identity/marker/SOS/
knowledge layers added on top. It has **not** been compiled — this was
written in a sandbox without an Android SDK/Gradle toolchain, so treat
everything below as "written in good faith against the real bitchat-android
source, not yet build-verified." Open it in Android Studio and expect a
first round of compile-error fixing, mostly around exact API names that
may have drifted from what's assumed here.

## What's real and wired end-to-end
- **Fork + rename**: `com.bitchat.android` → `com.kriptic.app` across all
  239 Kotlin files, gradle config, manifest, and app name.
- **Theme**: your `DesignTokens.kt`/`Color.kt`/`Type.kt` merged with
  bitchat's real dark/light/status-bar logic (`ui/theme/Theme.kt`).
- **Identity**: `identity/KripticIdentityRepository.kt` — immutable,
  once-only username stored via `EncryptedSharedPreferences`, gates the
  app on first launch (`identity/UsernameRegistrationScreen.kt`).
- **Channels**: `mesh/ChannelRegistry.kt` — the 4 default channels
  (General/Priority/Danger-Alert/Information), auto-joined on startup via
  bitchat's existing `joinChannel()`.
- **Markers**: `map/Marker*.kt` — Room-backed local cache, corroboration/
  aggregation logic (same-type pins within 75m from different reporters
  merge instead of duplicating), broadcast over the existing mesh channel-
  message pipeline (`map/MarkerBroadcastBridge.kt`) rather than a new
  binary packet type.
- **SOS**: `sos/SosPayload.kt` + `SosTrigger.kt` — no confirmation dialog,
  long-press-only trigger, same channel-envelope trick as markers.
- **Panic wipe**: `security/PanicWipeManager.kt`, hooked directly into
  bitchat's existing `ChatViewModel.panicClearAllData()` — extends it
  rather than replacing it, clears the marker DB and locked username.
- **Knowledge base**: `knowledge/*.kt` — Room FTS4 full-text search over
  the real `content/legal/know_your_rights.json` and
  `content/firstaid/first_aid.json` (currently placeholder content per
  their own `_readme` fields — genuinely not reviewed yet, and the app
  says so on the detail screen rather than hiding that).
- **Nav**: `ui/nav/KripticAppScaffold.kt` — 3-tab bottom nav (Messaging /
  Maps / Knowledge), the single integration seam replacing the old bare
  `ChatScreen()` call in `MainActivity.kt`.

## What's deliberately incomplete, and why
- **Map tiles**: real Delhi NCR data still doesn't exist (needs a
  Geofabrik extract this sandbox can't reach — see `map-data/README.md`).
  What's new this pass: a synthetic 16KB sample `.pmtiles` (rough NCR
  bbox + 5 fake labeled points, generated by `gen_sample_pmtiles.py`) IS
  now bundled and actually renders, via a small loopback HTTP bridge
  (`map/PmtilesLocalTileServer.kt` + a from-spec `map/PmtilesReader.kt`,
  since vanilla MapLibre Android has no `pmtiles://` support). This lets
  the whole pipeline — asset, local server, MapLibre style, marker
  overlay — be smoke-tested on a real device today, ahead of real map
  data existing. `PmtilesReader` was ported field-for-field from the
  reference Python `pmtiles` library and validated against files that
  library writes; not yet tested against planetiler/tippecanoe output.
- **Nostr/geohash removal**: the inherited internet-fallback code (17
  files, ~6,500 lines) is still physically present, just renamed. Fully
  stripping it is real surgery across core mesh files that needs compile
  verification at each step — safer to leave it in and flag it than to
  half-gut it blind. Recommend doing this as its own PR, file by file,
  with a working build at each commit.
- **SOS priority/TTL**: sends a 3x burst as a stand-in for real
  max-priority relay, since `MeshService`'s interface doesn't expose a
  per-message priority knob today. Documented in `SosTrigger.kt`.
- **Inter font**: falls back to system sans-serif; real `.ttf` files
  weren't fetchable from the build sandbox. Swap-in instructions are in
  `DesignTokens.kt`.
- **Old nickname-editing entry points**: bitchat's inherited settings/
  about UI may still let a user "change" their nickname elsewhere in the
  app, which would contradict Kriptic's immutable-username rule. This
  pass didn't hunt down every such entry point — worth an audit pass.
- **Content review**: both knowledge JSON files are explicitly
  placeholder pending a real medical/legal reviewer, per their own
  `_readme` fields. No content was invented or "filled in" here.
